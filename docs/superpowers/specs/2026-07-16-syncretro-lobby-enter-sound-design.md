# SyncRetro lobby enter_sound — design

**Date:** 2026-07-16
**Status:** approved, not yet implemented
**Scope:** `exec/load/syncretro_lobby.js`, both consoles' `syncretro.example.ini`,
`xtrn/syncivision/test_lobby_headless.js`

## Goal

Play a short, sysop-supplied sound when a player enters a SyncRetro console lobby
(SyncNES, SyncIvision) — the same one-shot entry sound SyncDuke and SyncDOOM
already play. Nothing ships; the sysop supplies the file.

## Why this shape

The mechanism already exists and is already shared. `exec/load/game_lobby.js`
exposes `enter_sound(door_dir, cfg)`, which SyncDuke and SyncDOOM each call in one
line from their `sd_main()`. It reads `cfg.lobby.enter_sound` (a file path, or a
wildcard it picks from at random per entry), probes the terminal, and plays the
file via `cterm_lib.js`'s SyncTERM audio APCs. `syncretro_lobby.js:44` already
loads `game_lobby.js` as `syncretro_lobby_gl`, so this adds no dependency.

## Decisions

### The sound is sysop-supplied only — no built-in

Considered and rejected: a synthesized entry blip via `cterm_lib.js`'s
`play_tone()` / `A;Synth`, which would need no asset and would work on terminals
that have the audio APC but no libsndfile (a strictly wider set than
`enter_sound` reaches).

Rejected on two grounds:

- **It cannot produce the sound anyone actually wants.** `A;Synth` takes shape,
  frequency and duration — there is no envelope. A console sound effect's
  character is largely its decay; flat tones that stop dead are not the same
  sound. Faking a decay by queueing descending-volume patches on one channel
  would be a zippered approximation, arguably worse than an honest beep.
- **Succeeding would be the problem, not the fix.** A Synth sequence tuned until
  it reproduces a specific commercial game's sound is a derivative of that sound;
  arriving as code rather than as a WAV changes nothing. Only a generic blip is
  clean, and a generic blip is not what the feature is for.

So SyncRetro matches its siblings: silent until a sysop configures a file.

### The key is `[lobby] enter_sound`, not `[audio] enter_sound`

`[lobby]` is `game_lobby.js`'s actual contract (`cfg.lobby.enter_sound`), so the
config object passes through with no remapping, and it matches SyncDuke and
SyncDOOM exactly.

SyncRetro's existing `[audio]` section is the wrong home despite being the
tempting one: it documents the **in-game Opus stream from the core** (`quality`,
`chunk_ms`, `prebuffer`, and a `volume` the player changes in-game with `+`/`-`).
A one-shot lobby SFX shares nothing with those, and filing it there would invite a
sysop to expect `volume` to govern it.

### Assets are not tracked

No audio ships, matching `xtrn/syncduke/quips/` (untracked) and the project's
stated discipline in `xtrn/syncnes/getroms.js` ("Being downloadable is not a
licence"). The console lobbies differ from their siblings in one respect worth
documenting honestly: Duke can tell a sysop to extract a quip from their own GRP
and DOOM to pull a sound from their own WAD, but a cartridge has no equivalent —
NES and Intellivision audio is generated by the APU/AY-3-8910 at runtime and does
not exist as data in the ROM. The example ini must not imply otherwise.

## Changes

### `exec/load/syncretro_lobby.js`

1. `syncretro_lobby_init()` currently reads only `iniGetObject("roms")` from the
   sysop's `syncretro.ini`. Add a `"lobby"` read, exposed so `syncretro_lobby()`
   can hand `enter_sound` the `{ lobby: { enter_sound: ... } }` it expects.
2. In `syncretro_lobby()`, after the BIOS-missing and no-cartridges checks and
   before the picker loop, call:

   ```js
   syncretro_lobby_gl.enter_sound(syncretro_lobby_dir, cfg);
   ```

   After the guards, so a fanfare never plays into an error message.

Both consoles inherit this from the one shared call site; no per-console lobby.js
change.

### `xtrn/syncnes/syncretro.example.ini`, `xtrn/syncivision/syncretro.example.ini`

A commented `[lobby]` section documenting `enter_sound`: path or wildcard,
relative to the door dir or absolute; formats (prefer OGG/WAV — see
`cterm_lib.js`'s note that MP3 needs libsndfile >= 1.1.0); that nothing ships and
copyrighted audio must not be redistributed; and that a terminal without
libsndfile hears nothing.

### `xtrn/syncivision/test_lobby_headless.js`

`fake_console` gains `write`, `inkey` and `clearkeybuffer` stubs. The harness runs
against the live install and reads the live `syncretro.ini`: today `enter_sound`
returns before `cterm_lib.js` is loaded, so the test passes — but the first time a
sysop sets the key, the harness would load `cterm_lib.js` and throw on the stub.
With the stubs, `query_fb`'s `inkey` returns `""`, the loop breaks, the probe
reports no audio, and the lobby stays silent. The test then passes whether or not
a sound is configured, rather than passing only until the feature is used.

Add an assertion that entering the lobby stays silent and launches no door under
the stub console — which is what the stubs exist to guarantee. Positive coverage
(a configured file actually reaching the terminal) is deliberately NOT automated:
it would mean writing a `[lobby]` key into the live `syncretro.ini` the harness
reads, and the harness must not edit a sysop's config. That case is covered by the
live check below.

## Error handling

All inherited from `enter_sound`, all silent:

| Condition | Behavior |
|---|---|
| no `[lobby] enter_sound` key | return before `cterm_lib.js` is loaded |
| path/wildcard matches no file | return |
| terminal has no libsndfile | return after the probe |
| file unreadable | return |

## Testing

- `jsexec /sbbs/xtrn/syncivision/test_lobby_headless.js` stays green — with and
  without `enter_sound` set in the live ini.
- Live: set `enter_sound` on SyncNES, enter the lobby from SyncTERM, hear it.
- Live negative: enter from a terminal without audio support; lobby paints
  normally and silently.

## Out of scope

- Any built-in or synthesized sound.
- Volume/pan/fade options on the entry sound (`enter_sound` passes no opts today;
  the siblings don't either — divergence would be gratuitous).
- The C door's in-game audio stack (`src/doors/termgfx/audio_mgr.c`), which is a
  parallel implementation of the same wire protocol and is not involved.

## Notes

A concern raised during design and **disproved**: that `enter_sound`'s fixed
`"lobby_enter"` name plus `cterm_lib.js`'s upload-once `_audio_stored` cache would
make a wildcard randomize only once per session. It does not — `enter_sound` does
`load({}, "cterm_lib.js")`, a fresh module scope per call, so `_audio_stored`
starts empty every entry. Verified under jsexec: two `load({}, ...)` calls yield
independent objects with independent caches. No fix is needed, in SyncRetro or in
SyncDuke.
