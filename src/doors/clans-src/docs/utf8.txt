[1.1]  Design decisions
-------------------------------------------------------------------------------
- CP437 is the default encoding (backward compatible with all existing files).
- UTF-8 is opt-in via a .u8. compound extension (e.g., strings.u8.txt).
- Devkit compilers and makepak detect .u8. files and convert UTF-8 to CP437
  at compile/pack time.  The game runtime is unchanged -- compiled output and
  pak contents remain CP437.
- Error on smart quotes and dashes (they break pipe/backtick colour codes).
- Accept known alternative Unicode codepoints for ambiguous CP437 mappings.
- Support the full CP437 codespace including control-character glyphs
  (0x01-0x1F, 0x7F).


[1.2]  File naming convention
-------------------------------------------------------------------------------
Given a source file foo.txt, the UTF-8 version is named foo.u8.txt.

The tool is invoked with the original name (foo.txt).  The u8_fopen() helper
probes for foo.u8.txt first; if found, it reads in UTF-8 mode.  If not found,
it falls back to foo.txt in CP437 mode (no conversion).

This applies to all extensions: .txt, .hlp, .asc, .ini, .evt.

If both files exist, the .u8. variant takes priority.  The original CP437
file can be kept as a reference or deleted.


[1.3]  Which tools are affected
-------------------------------------------------------------------------------
Devkit compilers (read text source, produce binary output):

    Tool        Source file     Input file(s)
    --------    -----------     -------------------
    langcomp    langcomp.c      strings.txt
    mcomp       mcomp.c         monsters.txt
    mitems      mitems.c        items.txt
    mspells     mspells.c       spells.txt
    mclass      mclass.c        classes.txt
    ecomp       ecomp.c         *.evt event scripts
    makenpc     makenpc.c       npcquote.txt

Pak builder (copies text files verbatim into pak):

    makepak     makepak.c       files listed in pak.lst

Not affected:

    chew        chew.c          Reads manifest + binary files only
    install     install.c       Reads .ini; possible future extension
    Game runtime                No changes; compiled output stays CP437

[2.3]  Control-character glyphs (32 entries, CP437 0x01-0x1F + 0x7F)
-------------------------------------------------------------------------------
In CP437, bytes 0x01-0x1F display as graphical glyphs rather than control
characters.  In UTF-8 source files, these glyphs are represented by their
Unicode equivalents.  rputs() does not intercept these bytes (only |, `, %,
\n); od_disp() sends them raw to the terminal.

    CP437   Unicode   Name
    -----   -------   ----
    0x01    U+263A    WHITE SMILING FACE
    0x02    U+263B    BLACK SMILING FACE
    0x03    U+2665    BLACK HEART SUIT
    0x04    U+2666    BLACK DIAMOND SUIT
    0x05    U+2663    BLACK CLUB SUIT
    0x06    U+2660    BLACK SPADE SUIT
    0x07    U+2022    BULLET
    0x08    U+25D8    INVERSE BULLET
    0x09    U+25CB    WHITE CIRCLE
    0x0A    U+25D9    INVERSE WHITE CIRCLE
    0x0B    U+2642    MALE SIGN
    0x0C    U+2640    FEMALE SIGN
    0x0D    U+266A    EIGHTH NOTE
    0x0E    U+266B    BEAMED EIGHTH NOTES
    0x0F    U+263C    WHITE SUN WITH RAYS
    0x10    U+25BA    BLACK RIGHT-POINTING POINTER
    0x11    U+25C4    BLACK LEFT-POINTING POINTER
    0x12    U+2195    UP DOWN ARROW
    0x13    U+203C    DOUBLE EXCLAMATION MARK
    0x14    U+00B6    PILCROW SIGN
    0x15    U+00A7    SECTION SIGN
    0x16    U+25AC    BLACK RECTANGLE
    0x17    U+21A8    UP DOWN ARROW WITH BASE
    0x18    U+2191    UPWARDS ARROW
    0x19    U+2193    DOWNWARDS ARROW
    0x1A    U+2192    RIGHTWARDS ARROW
    0x1B    U+2190    LEFTWARDS ARROW
    0x1C    U+221F    RIGHT ANGLE
    0x1D    U+2194    LEFT RIGHT ARROW
    0x1E    U+25B2    BLACK UP-POINTING TRIANGLE
    0x1F    U+25BC    BLACK DOWN-POINTING TRIANGLE
    0x7F    U+2302    HOUSE

Note: raw control bytes (0x01-0x1F, 0x7F) in a UTF-8 file pass through
as-is.  Only the Unicode glyph equivalents listed above trigger conversion.
This means 0x09 (tab), 0x0A (newline), 0x0D (carriage return) work normally
in UTF-8 files -- they are NOT converted to white circle / inverse white
circle / eighth note.


[2.4]  Alternative codepoints accepted silently
-------------------------------------------------------------------------------
These Unicode codepoints are visual or semantic equivalents of existing
CP437 mappings.  The converter accepts them without warning, mapping to the
same CP437 byte as the canonical codepoint.

    Alternative   Canonical   CP437   Note
    -----------   ---------   -----   ----
    U+03B2        U+00DF      0xE1    Greek beta -> sharp-s
    U+03BC        U+00B5      0xE6    Greek mu -> micro sign
    U+2205        U+03C6      0xED    Empty set -> phi
    U+2208        U+03B5      0xEE    Element of -> epsilon
    U+2126        U+03A9      0xEA    Ohm sign -> omega
    U+00D8        U+03C6      0xED    O-stroke -> phi (visual similarity)
    U+03D5        U+03C6      0xED    Phi symbol -> phi
    U+2300        U+03C6      0xED    Diameter sign -> phi


[2.5]  Blocked codepoints (error with helpful message)
-------------------------------------------------------------------------------
These Unicode codepoints could plausibly appear via copy-paste from word
processors.  They are blocked because they would break the game's pipe-code
(|XX) and backtick-code (`XY) colour formatting.

    Codepoint   Error message
    ---------   -------------
    U+2018      use ASCII apostrophe ' instead
    U+2019      use ASCII apostrophe ' instead
    U+201C      use ASCII quotation mark " instead
    U+201D      use ASCII quotation mark " instead
    U+2014      use -- (double hyphen) instead
    U+2013      use - (hyphen) instead


Implementation
===============================================================================

[3.1]  Conversion rules
-------------------------------------------------------------------------------
When a .u8. file is detected, the converter processes each line after
fgets():

1. Bytes 0x00-0x7F: pass through unchanged (identical in ASCII/UTF-8/CP437).
   This includes tabs (0x09), newlines (0x0A), and carriage returns (0x0D).

2. Multi-byte UTF-8 sequences (leading byte 0xC0-0xF4): decode the Unicode
   codepoint, look up in the reverse table.
   - Found in canonical or alternative table -> emit single CP437 byte.
   - Found in blocked table -> print error with suggestion, abort.
   - Not found -> print error ("U+XXXX has no CP437 equivalent"), abort.

3. Invalid UTF-8 (unexpected continuation byte 0x80-0xBF without a leading
   byte, truncated sequence, overlong encoding, codepoints above U+10FFFF)
   -> print error ("invalid UTF-8 at byte offset N"), abort.

The conversion always shrinks or preserves buffer length (multi-byte UTF-8
sequences become single CP437 bytes), so it can be done in-place.


[4.1]  FreeBSD / tooling notes
-------------------------------------------------------------------------------
Commands and tools that work correctly on this platform:

    LANG=C ggrep -cP '[\x80-\xFF]' <file>
        Count lines containing CP437 high bytes.
        MUST use LANG=C to prevent UTF-8 locale from masking matches.
        MUST use ggrep (GNU grep), not system grep (BSD, lacks -P).

    python3 -c "open('file','rb').read()"
        Read/write CP437 files safely.  Always use 'rb'/'wb' mode.

    NEVER use Claude Code Read/Edit/Write tools on CP437 files.
        They interpret as UTF-8 and corrupt every byte >= 0x80 to U+FFFD
        (3-byte sequence 0xEF 0xBF 0xBD).  This has caused data loss.

    Claude Code Read tool strips UTF-8 BOM (0xEF 0xBB 0xBF).
        BOM-based encoding detection is unreliable.  Use .u8. file
        extension convention instead.
