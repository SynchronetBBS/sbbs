[1.1]  Charisma stat has no combat or empire effect
-------------------------------------------------------------------------------
The {Cyy} ACS condition gives Charisma a role in quest events and NPC
dialogue -- PAK developers can gate content on the clan leader's
Charisma score.  However, ATTR_CHARISMA is still never read in fight.c
or empire.c, so it contributes nothing to combat or empire outcomes.

Options: give it a mechanical role in combat or empire (e.g. NPC loyalty,
trading prices, recruitment cost), or accept that its role is limited to
quest/dialogue gating via {C}.

Suggestion: use Charisma to break vote ties for village ruler.  When
ChooseNewLeader() in voting.c finds multiple candidates tied for the
most votes, compare the clan leaders' Charisma scores instead of
choosing at random.  If the leaders' Charisma values also tie, compare
the next clan member, and so on down the roster.  Only fall back to
my_random() if every member's Charisma matches across all tied clans.

[1.2]  Clean up mstrings.h GNUmakefile dependencies
-------------------------------------------------------------------------------
The mstrings.h target in src/GNUmakefile and the native-devkit dependency in
data/GNUmakefile both grew organically to solve the chicken-and-egg problem of
langcomp needing to exist before mstrings.h can be generated.  The current
approach works but could be cleaner -- the NATIVEEXEFILE computation and the
filter-out workaround for shared langcomp objects are worth revisiting.

[1.4]  install: rename prompt limited to 13 characters
-------------------------------------------------------------------------------
When the user chooses "Rename" during an overwrite conflict, DosGetStr
limits the new name to 13 characters (MAX_FILENAME_LEN) -- the old DOS
8.3 convention.  This prevents renaming to any long filename.
Source: install.c lines 349, 741.

[1.5]  install: silent truncation of GUM and INI filenames
-------------------------------------------------------------------------------
The GUM filename (! line) and INI filename (command argument) are stored
in 25-byte buffers and silently truncated via strlcpy if they exceed 24
characters.  Manifest entries beyond the 75th are also silently ignored.
These should produce warnings.
Source: install.c lines 51, 147, 1023.

[1.7]  Village types defined but never assigned
-------------------------------------------------------------------------------
defines.h declares three village types (V_WOODLAND=1, V_COASTAL=2,
V_WASTELAND=3) and items can be tagged with a VillageType so they only
appear in matching villages (items.c:798).  However, Village.Data.VillageType
is never set -- not by readcfg.c, config.c, reset.c, village.c, or any
other code path.  It stays at its struct-zero default (0 == V_ALL), so items
with VillageType 1-3 can never appear in the market.  The separate
village_data.TownType field is always hardcoded to -1 ("unknown for now").

The type names "Woodland", "Coastal", "Wasteland" come only from the
#define identifiers; they appear nowhere in any user-facing string, menu,
or documentation.

Options: (a) finish the feature by adding a VillageType config keyword
(or per-BBS assignment in InterBBS setup) so sysops can designate village
types and the item filtering becomes meaningful; (b) remove the dead
defines, the struct field, and the items.c filter if the feature is not
wanted.

[1.8]  Dead and orphaned help-file topics
-------------------------------------------------------------------------------
Two topics in menus.hlp are never displayed -- no Help() or GetChoice() call
references them, and menus.hlp has no GeneralHelp() path:

    ^Eliminated     "You have been eliminated" screen; the game uses
                    DeleteClan() and a news message instead.
    ^Special        Promotional "door sale" copy from the original author;
                    never referenced.

Three topics in clans.hlp have no direct Help() call but remain browsable
through the MainHelp > General Help menu (help.c GeneralHelp(ST_CLANSHLP)):

    ^Newbies        Brief welcome message for new players.
    ^Colours        Pipe-code colour instructions.
    ^Beta Note      Bug-report URLs and feature wishlist; content is stale
                    (dead URLs, features from the original roadmap).

Options: (a) delete the two dead menus.hlp topics; (b) wire ^Eliminated
into the elimination code path so it is actually shown; (c) update the
stale clans.hlp topics or remove the ones that duplicate information
available elsewhere.

[1.9]  UTF-8 source file support for devkit compilers
-------------------------------------------------------------------------------
Add a shared u8cp437.c module with a reverse lookup table (Unicode -> CP437),
u8_fopen() to probe for .u8. variant files, and u8_fgets() to convert UTF-8
lines to CP437 on read.  Modify 7 devkit compilers (langcomp, mcomp, mitems,
mspells, mclass, ecomp, makenpc) and makepak to auto-detect UTF-8 source
files and convert to CP437 at compile/pack time.  Game runtime unchanged.
Full design reference in docs/utf8.txt.

[1.10]  Convert stock CP437 data files to UTF-8
-------------------------------------------------------------------------------
Done.  All 10 stock CP437 text files converted to UTF-8 with .u8. naming.
The 7 data/ files are wired into the build (data/GNUmakefile, data/pak.lst);
the 3 installer .ini files are created but not wired in (see [1.11]).
Binary output verified byte-identical (strings.xl, npcquote.q, spells.spl,
clans.npc, clans.pak, mstrings.h).  Build system updated: data/GNUmakefile,
data/pak.lst, root GNUmakefile devkit target, devkit/genall.mak,
devkit/genall.bat.  Original CP437 files kept in repo for reference.

[1.11]  UTF-8 support for install.c
-------------------------------------------------------------------------------
Extend .u8. file detection to install.c, which reads .ini files containing
CP437 (installer/install.ini has 1 CP437 line, installerdk/clandev.ini has
13).  Deferred from [1.9] because install is a standalone tool, not a devkit
compiler.
