===============================================================================

[1.1]  Charisma stat has no effect
-------------------------------------------------------------------------------
struct pc carries a Charisma attribute that can be set by class definitions
(mclass.c), items (mitems.c), and spells (mspells.c), and is shown in stat
displays.  It is never read in fight.c or empire.c, so it contributes nothing
to combat or empire outcomes.  Noted in progress.log as "charisma is useless!"

Options: give it a mechanical role (e.g. NPC loyalty, trading prices, odds of
seeing wandering NPCs, recruitment cost), or remove it from the data files and
display if there is no intention to use it.

[1.2]  Clean up mstrings.h GNUmakefile dependencies
-------------------------------------------------------------------------------
The mstrings.h target in src/GNUmakefile and the native-devkit dependency in
data/GNUmakefile both grew organically to solve the chicken-and-egg problem of
langcomp needing to exist before mstrings.h can be generated.  The current
approach works but could be cleaner -- the NATIVEEXEFILE computation and the
filter-out workaround for shared langcomp objects are worth revisiting.

[1.3]  stats.hlp Charisma description outdated
-------------------------------------------------------------------------------
stats.hlp says Charisma is "currently only needed for speaking out for or
against the ruler."  The speaking-out/approval system was removed in v0.89b.
Charisma is tracked in struct pc and can be raised by training, items, and
spells, but ATTR_CHARISMA is never read by any current gameplay code.

The fix depends on the resolution of [1.1]: if Charisma gets a mechanical
role, update the description accordingly; if it is removed, remove it from
the help file and stat display.  In the interim, the false "speaking out"
claim should be replaced with a note that the stat is defined but currently
has no active effect.

[1.4]  install: rename prompt limited to 13 characters
-------------------------------------------------------------------------------
When the user chooses "Rename" during an overwrite conflict, DosGetStr
limits the new name to 13 characters (MAX_FILENAME_LEN) -- the old DOS
8.3 convention.  This prevents renaming to any long filename.
Source: install.c lines 349, 741.

[1.5]  install: silent truncation of GUM and INI filenames
-------------------------------------------------------------------------------
The GUM filename (! line) and INI filename (command argument) are stored
in 25-byte buffers and silently truncated via strlcpy if they exceed 24
characters.  Manifest entries beyond the 75th are also silently ignored.
These should produce warnings.
Source: install.c lines 51, 147, 1023.

[1.7]  Village types defined but never assigned
-------------------------------------------------------------------------------
defines.h declares three village types (V_WOODLAND=1, V_COASTAL=2,
V_WASTELAND=3) and items can be tagged with a VillageType so they only
appear in matching villages (items.c:798).  However, Village.Data.VillageType
is never set -- not by readcfg.c, config.c, reset.c, village.c, or any
other code path.  It stays at its struct-zero default (0 == V_ALL), so items
with VillageType 1-3 can never appear in the market.  The separate
village_data.TownType field is always hardcoded to -1 ("unknown for now").

The type names "Woodland", "Coastal", "Wasteland" come only from the
#define identifiers; they appear nowhere in any user-facing string, menu,
or documentation.

Options: (a) finish the feature by adding a VillageType config keyword
(or per-BBS assignment in InterBBS setup) so sysops can designate village
types and the item filtering becomes meaningful; (b) remove the dead
defines, the struct field, and the items.c filter if the feature is not
wanted.

[1.8]  Add Charisma ACS condition for events and NPC chat
-------------------------------------------------------------------------------
This would give the Charisma attribute ([1.1]) a gameplay role without
touching the combat or empire code.

The ACS evaluator in quests.c legal() already supports conditions for
flags (G/H/P/D/T), quest completion (Q), gold ($), mine level (L/K),
and random chance (R).  Add a new condition letter -- for example C --
that tests the clan leader's Charisma score:

    {C20}   true if leader's Charisma >= 20
    {!C50}  true if leader's Charisma < 50

Implementation sketch:

    case 'C' :  // leader Charisma >= N
        pcCurrentPos++;
        iTemp = 0;
        while (isdigit(*pcCurrentPos))
            szIndex[iTemp++] = *(pcCurrentPos++);
        szIndex[iTemp] = 0;
        UseCurrent = true;
        iTemp = ato16(szIndex, "Charisma", __func__);
        bCurrent =
            (PClan.Member[0]->Attributes[ATTR_CHARISMA] >= iTemp);
        break;

This follows the same pattern as the existing K (min mine level) case.
PClan.Member[0] is the clan leader; a NULL check should be added for
safety.

Beyond events, ChatVillagers() in npc.c could also use the leader's
Charisma to influence NPC visibility -- e.g. an NPC only appears in the
chat list when Charisma exceeds a threshold stored in NPCInfo.  This
would make the stat meaningful in everyday play: higher Charisma means
access to more NPCs and their topics.

PAK developers would then be able to write quests and NPCs that reward
investing in Charisma, resolving [1.1] without any combat formula
changes.  The stats.hlp description ([1.3]) could be updated to say
Charisma affects NPC encounters and quest outcomes.

Affected files: quests.c (legal), npc.c (ChatVillagers), structs.h
(NPCInfo if adding a Charisma threshold field), clandev.txt (document
the new ACS letter), stats.hlp (update description).
