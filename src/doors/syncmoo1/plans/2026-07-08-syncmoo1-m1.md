# syncmoo1 M1 (Playable Vertical Slice) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:subagent-driven-development (recommended) or
> superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** A `syncmoo1` door binary that launches from Synchronet over
SyncTERM, renders 1oom's Master of Orion 1 main-menu / first screen as sixel,
and is navigable with keyboard + (cell-resolution) mouse.

**Architecture:** Vendor the frozen `fork1oom/1oom` tree; add a new 1oom `hw`
backend (`hw_sbbs.c`) whose `main()` does BBS/door setup then calls
`main_1oom()`. The backend implements 1oom's `hw.h` by delegating rendering to
`termgfx` (sixel) and injecting decoded terminal input into 1oom's global
input state (`kbd_add_keypress`, `mouse_set_xy_from_hw`). One door-owned
CMakeLists compiles 1oom's `game/` + `ui/classic/` + `os/unix/` + our glue +
`termgfx` into a single binary. See [../DESIGN.md](../DESIGN.md).

**Tech Stack:** C (C11), CMake, `termgfx` (sixel/JXL/text + APC + pace +
sbbs_node), xpdev (sockwrap/ini), vendored 1oom (GPLv2+), DOOR32.SYS.

## Global Constraints

- **Canvas:** 1oom renders native **320×200, 8-bit palettized**. Never assume
  8×16 cells; use termgfx's probed cell size.
- **Never edit the vendored `1oom/` tree.** All door logic lives in
  `hw_sbbs.c` + `syncmoo1_*.c`. Absorb engine warnings via CMake flags
  (`-w -fcommon`), never by patching upstream.
- **Never modify `../termgfx`** without checking the other consumers first
  (per `termgfx/CLAUDE.md`).
- **House style** (`../../uncrustify.cfg`) applies to OUR files only.
- **termgfx is I/O-free:** every encoder writes into a caller buffer; the door
  owns the socket + read/poll loop.
- **Binary name:** `syncmoo1`. **`idstr_hw = "sbbs"`.**
- **Engine upstream:** `fork1oom/1oom`, record ref/date/license in
  `PROVENANCE.md`.
- **No CMake `install()`** — deploy via `deploy.js`; build never touches the
  live install.
- 1oom requires original MoO1 (v1.3) LBX data files at runtime; provided per
  install, out of scope for code.

## Resolved integration facts (from reading 1oom source)

- **Entry:** the hw backend defines `int main(int,char**)` → `main_1oom()`
  (`src/main.c:62`). We do door setup in our `main()` first.
- **hw interface:** `src/hw.h` — full contract. Minimum functional set modeled
  on `src/hw/sdl/*` and `src/hw/nop/hwnop.c`.
- **Video buffers:** 1oom expects the backend to own 4 pages of
  `bufw*bufh` indexed bytes. `hw_video_get_buf()` returns the back buffer;
  `hw_video_draw_buf()` presents + flips + returns the new back buffer;
  `hw_video_copy_back_to/from_page2/3()` are `memcpy`s (see
  `hw/sdl/hwsdl_video.c:55-105`). The game DRIVES these — we write no loop.
- **Palette:** `hw_video_set_palette(pal,first,num)` must call
  `ui_palette_set(pal,first,num)` (from `ui_sources/palette.c`) and store our
  own 256×RGB copy for sixel.
- **Keyboard injection:** `kbd_add_keypress(mookey_t key, uint32_t mod, char c)`
  and `kbd_set_pressed(key, mod, bool)` (`src/kbd.h:210,213`). Key enum
  `mookey_t` is ASCII-mapped for printables; mods `MOO_MOD_{SHIFT,ALT,CTRL}`.
- **Mouse injection (absolute):** set 1oom globals via
  `mouse_set_xy_from_hw(x,y)`, `mouse_set_buttons_from_hw(mask)` (mask =
  `MOUSE_BUTTON_MASK_LEFT|RIGHT`), `mouse_set_scroll_from_hw(s)`
  (`src/mouse.h:21-23`). 1oom draws its OWN cursor from `moouse_x/y` — we do
  not composite a cursor and do not rely on the terminal pointer.
  `hw_mouse_set_xy()` (game→hw pointer warp) is a **no-op** for a terminal.
- **Build:** modular static libs. Executable = top-level `util_sources` +
  `audio_sources` + `ui_sources` (from `src/Makefile.am:34-95`) + `game/`
  (`libgame`+`libgamemain`) + `ui/classic/` + `os/unix/os.c` + our hw backend.
  `config.h` and `version.inc` are autotools-generated — freeze them (Task 1).

### Task 4: `syncmoo1_map` — mouse cell→game mapping (TDD, pure)

The syncconquer pattern: SGR report → cell-center in canvas pixels → clamp to
image rect → rescale to 320×200.

**Files:**
- Modify: `syncmoo1_map.h`, `syncmoo1_map.c`
- Modify: `tests/test_map.c`

**Interfaces:**
- Produces:
```c

                 int cw, ch;              /* real probed cell size */
                 int pixel_mode; } sm_geom_t;
/* col,row are 1-based SGR coords. Writes game coords in [0,319]x[0,199]. */
void sm_map_mouse(const sm_geom_t *g, int col, int row, int *gx, int *gy);
```
- Consumes: nothing engine-side (pure).

- [ ] **Step 1: Write failing tests** (append to `tests/test_map.c`): with a
  640×400 image at offset (0,0), 8×16 cells, cell-mode: col=1,row=1 →
  gx≈`(0*8+4)*320/640`=2, gy≈`(0*16+8)*200/400`=4; a mid-canvas cell maps
  proportionally; out-of-rect clamps to 0..319 / 0..199; pixel-mode uses
  `col-1` directly.

```c
sm_geom_t g = { .ew=640,.eh=400,.dx=0,.dy=0,.cw=8,.ch=16,.pixel_mode=0 };
int gx, gy;
sm_map_mouse(&g, 1, 1, &gx, &gy);   assert(gx==2 && gy==4);
sm_map_mouse(&g, 80, 25, &gx, &gy); assert(gx>=317 && gy>=195);
g.pixel_mode = 1;
sm_map_mouse(&g, 321, 201, &gx, &gy); assert(gx==160 && gy==100);
```

- [ ] **Step 2: Run to verify FAIL.**

- [ ] **Step 3: Implement `sm_map_mouse`** (port of syncconquer
  `door_input.c:299-339`):

```c
void sm_map_mouse(const sm_geom_t *g, int col, int row, int *gx, int *gy) {
    int px, py;
    if (g->pixel_mode) { px = col - 1; py = row - 1; }
    else { px = (col-1)*g->cw + g->cw/2; py = (row-1)*g->ch + g->ch/2; }
    if (px < g->dx) px = g->dx; if (px > g->dx+g->ew-1) px = g->dx+g->ew-1;
    if (py < g->dy) py = g->dy; if (py > g->dy+g->eh-1) py = g->dy+g->eh-1;
    int x = (px - g->dx) * 320 / g->ew;
    int y = (py - g->dy) * 200 / g->eh;
    if (x < 0) x = 0; if (x > 319) x = 319;
    if (y < 0) y = 0; if (y > 199) y = 199;
    *gx = x; *gy = y;
}
```
